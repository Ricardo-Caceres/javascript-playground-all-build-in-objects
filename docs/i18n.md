# Internationalization (i18n)

This document covers how i18n is implemented in the JavaScript Playground and how to extend it.

---

## Overview

| | |
|---|---|
| Library | [next-intl v4](https://next-intl.dev/) |
| Routing | URL-based — `/en/*`, `/es/*` |
| Default locale | `en` (redirects from `/`) |
| Supported locales | `en`, `es` |
| Exercise translations | `messages/exercises/<locale>.json` |

---

## Architecture

### Locale routing

All routes live under `src/app/[locale]/`. The `[locale]` segment is validated by middleware against the list in `src/i18n/routing.ts`. Unknown locales fall through to a 404.

```
/              → 307 → /en
/en            → Home (English)
/es            → Home (Spanish)
/en/exercises/array/array-map-1   → Exercise page (English)
/es/exercises/array/array-map-1   → Exercise page (Spanish)
```

### Key files

| File | Purpose |
|------|---------|
| `src/i18n/routing.ts` | Defines `locales` and `defaultLocale` |
| `src/i18n/navigation.ts` | Locale-aware `Link`, `useRouter`, `usePathname`, `redirect` |
| `src/i18n/request.ts` | Server-side message loader (`getRequestConfig`) |
| `src/i18n/exerciseTranslations.ts` | Merges exercise data with locale-specific translations |
| `middleware.ts` | next-intl middleware — handles locale detection and redirects |
| `messages/en.json` | UI strings in English |
| `messages/es.json` | UI strings in Spanish |
| `messages/exercises/es.json` | Spanish translations for all 1,986 exercises |

### UI translations

UI strings are namespaced in `messages/<locale>.json`:

```json
{
  "nav": { "home": "Home", "stats": "Stats" },
  "home": { "title": "JavaScript Playground", "subtitle": "..." },
  "exercise": { "run": "Run", "reset": "Reset", "hint": "Hint" },
  "search": { "placeholder": "Search exercises..." }
}
```

Components consume them with `useTranslations`:

```tsx
const t = useTranslations('nav')
return <Link href="/">{t('home')}</Link>
```

### Exercise translations

Exercise content (title, description, hints, test descriptions) is stored separately from the exercise data files, in `messages/exercises/<locale>.json`, keyed by slug:

```json
{
  "array-map-1": {
    "title": "Duplicar cada elemento con map",
    "description": "Usa `Array.prototype.map` para...",
    "hints": ["Recuerda que map devuelve un nuevo array"],
    "tests": [{ "description": "Debería devolver [2, 4, 6]" }]
  }
}
```

At render time, `getLocalizedExercise(exercise, locale)` in `src/i18n/exerciseTranslations.ts` merges the base exercise with the locale translation, falling back to English if the translation is missing.

## Language Switcher

`src/shared/components/LanguageSwitcher.tsx` is a client component that uses `useLocale()` and `router.replace(pathname, { locale })` to switch locale without a full page reload. It reads the available locales dynamically from `routing.locales`.

## Translation Script

`scripts/translate-exercises.mjs` uses Claude (Anthropic API) to translate all exercise content.

### Usage

```bash
ANTHROPIC_API_KEY=sk-ant-... node scripts/translate-exercises.mjs [--locale es] [--batch 20]
```

- `--locale` — target locale (default: `es`)
- `--batch` — exercises per API call (default: `20`)
- **Idempotent** — already-translated slugs are skipped; safe to re-run
- Saves progress after every batch (crash-safe)

### Cost estimate

Uses `claude-haiku-4-5`. For 1,986 exercises: approximately **$0.10–0.20**.

## Adding a new language

1. **Register the locale** in `src/i18n/routing.ts`:
   ```ts
   export const routing = defineRouting({
     locales: ['en', 'es', 'fr'],  // add here
     defaultLocale: 'en',
   })
   ```

2. **Create UI messages** — copy `messages/en.json` to `messages/fr.json` and translate the values.

3. **Run the exercise translation script**:
   ```bash
   ANTHROPIC_API_KEY=sk-ant-... node scripts/translate-exercises.mjs --locale fr --batch 20
   ```
   This creates `messages/exercises/fr.json` with all 1,986 exercises translated.

4. **Update `exerciseTranslations.ts`** to import and map the new locale:
   ```ts
   import frExercises from '../../messages/exercises/fr.json'
   const translationMap: Record<string, ExerciseTranslations> = { es: esExercises, fr: frExercises }
   ```

## Locale detection

next-intl detects the locale from:
1. The URL path prefix (`/en`, `/es`)
2. Falls back to `defaultLocale` (`en`) if no match

There is no cookie or `Accept-Language` header negotiation — the URL is always the source of truth.
